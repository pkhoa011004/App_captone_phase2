# Datapack Insights For CDO - TF1 Triage Hub

Owner: AI team TF1
Status: CDO handoff companion
Last updated: 2026-06-25

## Purpose

This note explains the business meaning of the TF1 datapack, not just the file locations. CDO teams can use it to understand what each RCAEval-derived case is meant to represent, what data needs to be hosted, what the AI should conclude, and what evidence should be visible in the report UI.

The primary scenario source is the RCAEval adapted subset under:

```text
capstone/tf-1/ai/engine-skeleton/datapack/external/
```

The checked-in raw selected-case subset contains RCAEval `metrics.json` and `inject_time.txt`. Full RCAEval `logs.csv` and `traces.csv` files are intentionally not committed because some selected files exceed GitHub's normal file size limit. The adapted requests and evidence bundles include bounded snippets from the exact selected RCAEval case files when those files exist. The `data_lineage` block in each bundle marks which records are selected-case RCAEval telemetry and which records are TF1 supplemental operational context.

## How CDO Should Read The Datapack

| Layer | Path | Meaning |
|---|---|---|
| Raw selected subset | `datapack/external/rcaeval-subsets/` | Source RCAEval case metrics and injection timestamps currently checked into the repo. Exact raw logs/traces should be downloaded or hosted externally because the upstream files are large. |
| Adapted triage requests | `datapack/external/adapted/` | `/v1/triage` request payloads generated from selected RCAEval cases. Logs/traces are present only when the exact selected case provides them. |
| Evidence bundles | `datapack/external/evidence-bundles/` | CDO-hostable records combining metrics, query hints, logs, traces, deploys, ownership, runbooks, and lineage. |
| E2E report output | `reports-e2e/rcaeval-*.json` | Report JSON generated by posting adapted RCAEval requests into the local API. |

CDO does not need to infer root cause from file names. Each case name follows:

```text
{dataset}_{service}_{fault}_{run}
```

Examples:

- `re2tt_ts-auth-service_loss_1`: RE2 Train Ticket case, `ts-auth-service`, loss/unavailability fault, first run.
- `re2ss_catalogue_delay_2`: RE2 Sock Shop case, `catalogue`, latency/delay fault, second run.
- `re1ss_user_cpu_3`: RE1 Sock Shop case, `user`, CPU resource fault, third run.

## Scenario Stories

### 1. Critical Service Down

These cases represent high-impact outage or service-unavailable incidents. The expected AI behavior is `DIAGNOSED / critical_service_down`, with owner escalation and service-down runbook checks. The AI must not auto-restart or auto-rollback.

| Case | Incident time | Service | What happened | Evidence meaning | Expected output |
|---|---|---|---|---|---|
| `re1ss_user_loss_2` | 2023-08-27 18:54:58 UTC | `user` | Sock Shop `user` service is treated as unavailable from a `loss` fault. | Metrics include `user_cpu`, `user_latency-*`, `user_workload`, and related `user-db` signals. This RE1 selected case is metrics-only; logs/traces remain empty. | `DIAGNOSED`, `critical_service_down`, confidence about `0.86`, top candidate `user`. |
| `re2ss_orders_loss_1` | 2024-01-18 19:43:30 UTC | `orders` | Sock Shop `orders` service is treated as unavailable from a `loss` fault. | Metrics include `orders_error`, `orders_latency-*`, `orders_socket`, `orders-db_*`. The bundle also includes bounded snippets from the exact selected-case RCAEval `logs.csv`; no selected-case trace file exists. | `DIAGNOSED`, `critical_service_down`, confidence about `0.86`, top candidate `orders`. |
| `re2tt_ts-auth-service_loss_1` | 2024-01-23 04:10:03 UTC | `ts-auth-service` | Train Ticket auth service is down or critically degraded from a `loss` fault. | Metrics include `ts-auth-service_error`, `ts-auth-service_diskio`, latency, socket, workload. The bundle includes bounded snippets from exact selected-case RCAEval `logs.csv` and `traces.csv`. | `DIAGNOSED`, `critical_service_down`, confidence about `0.86`, top candidate `ts-auth-service`. |

What CDO should host or expose:

- Metric window around the incident time.
- Related service/dependency metrics listed in `query_hints.metrics`.
- Bounded outage logs and trace summaries from the evidence bundle.
- Ownership and runbook records so Slack/Jira routing can be shown.

### 2. Latency Degradation

These cases represent high-latency incidents where the service still exists but response time or dependency path is degraded. The expected AI behavior is `DIAGNOSED / latency_degradation`, with dependency/runbook investigation and human-reviewed rollback consideration only when deploy correlation is present.

| Case | Incident time | Service | What happened | Evidence meaning | Expected output |
|---|---|---|---|---|---|
| `re1ob_cartservice_delay_3` | 2023-08-21 08:59:08 UTC | `cartservice` | Online Boutique cart service has a delay fault. | Metrics include `cartservice_latency-*`, `cartservice_cpu`, `cartservice_mem`, and related service latency. This RE1 selected case is metrics-only. | `DIAGNOSED`, `latency_degradation`, confidence about `0.82`, top candidate `cartservice`. |
| `re1ss_carts_delay_1` | 2023-08-26 17:04:06 UTC | `carts` | Sock Shop carts service has a delay fault. | Metrics include `carts_latency-*`, `carts_error`, `carts-db_*`, and neighboring catalogue signals. This RE1 selected case is metrics-only. | `DIAGNOSED`, `latency_degradation`, confidence about `0.82`, top candidate from evidence, often `carts-db`. |
| `re2ss_catalogue_delay_2` | 2024-01-20 00:18:03 UTC | `catalogue` | Sock Shop catalogue service has a delay fault. | Metrics include `catalogue_latency-*`, `catalogue_diskio`, `catalogue_socket`, `catalogue-db_*`. The bundle also includes bounded snippets from the exact selected-case RCAEval `logs.csv`; no selected-case trace file exists. | `DIAGNOSED`, `latency_degradation`, confidence about `0.82`, top candidate `catalogue`. |

What CDO should host or expose:

- Latency and dependency metric series.
- Log snippets matching latency, timeout, dependency, or SLO language.
- Trace summaries that show slow downstream spans or bottleneck service.
- Deploy metadata if CDO wants to demo "recent change may have contributed".

### 3. Noisy Or Low-Confidence Alert

These cases represent resource anomalies that should not be treated as confirmed outages. The expected AI behavior is `INVESTIGATE / noisy_or_ambiguous_alert`, confidence about `0.45`, and observe or human-review actions only.

| Case | Incident time | Service | What happened | Evidence meaning | Expected output |
|---|---|---|---|---|---|
| `re1ob_cartservice_mem_1` | 2023-05-28 11:25:33 UTC | `cartservice` | Online Boutique cart service has a memory-resource anomaly. | Metrics include `cartservice_mem`, `cartservice_cpu`, `cartservice_latency`, plus neighboring memory signals. It is intentionally mapped to low severity for TF1 noisy-alert behavior. | `INVESTIGATE`, `noisy_or_ambiguous_alert`, confidence about `0.45`, top candidate `cartservice`. |
| `re1ss_user_cpu_3` | 2023-08-27 16:10:48 UTC | `user` | Sock Shop user service has a CPU-resource anomaly. | Metrics include `user_cpu`, `user-db_cpu`, latency, and memory. The intended response is not owner paging as a hard outage unless other signals agree. | `INVESTIGATE`, `noisy_or_ambiguous_alert`, confidence about `0.45`, top candidate `user`. |
| `re1tt_ts-route-service_disk_3` | 2023-12-12 15:46:33 UTC | `ts-route-service` | Train Ticket route service has a disk-resource anomaly. | Metrics include `ts-route-service_error`, latency, memory, workload, and related Train Ticket service metrics. Treat as noisy/ambiguous until user impact is confirmed. | `INVESTIGATE`, `noisy_or_ambiguous_alert`, confidence about `0.45`, top candidate `ts-route-service`. |

What CDO should host or expose:

- Resource metrics and any user-impacting metrics needed to confirm or reject impact.
- One or more bounded "recovered/retry" log snippets, if using the evidence bundle.
- Trace summaries that show no confirmed user-impacting failure.
- No deploy event is required for noisy scenarios unless CDO has one.

## Data Lineage And Caveats

Current checked-in adapted requests have RCAEval logs/traces only where the selected case provides them:

- `re2ss_orders_loss_1`: metrics + 20 selected-case log snippets + no traces.
- `re2ss_catalogue_delay_2`: metrics + 20 selected-case log snippets + no traces.
- `re2tt_ts-auth-service_loss_1`: metrics + 20 selected-case log snippets + 20 selected-case trace spans.
- RE1 selected cases: metrics only; logs and traces are empty by dataset availability.

Evidence bundles therefore have:

- selected-case RCAEval metrics and injection timestamp,
- selected-case RCAEval logs/traces when the case includes `logs.csv` or `traces.csv`,
- TF1 supplemental deploy metadata, ownership, and runbooks.

Operational records are supplemental by design:

- RCAEval does not provide deploy metadata.
- RCAEval does not provide ownership routing.
- RCAEval does not provide TF1 runbooks.

CDO should therefore treat selected-case RCAEval telemetry as the primary RCA signal, and treat deploy/ownership/runbook as TF1 operational scaffolding until CDO connects real service catalog, CI/CD event, Jira, or Confluence sources.

## What CDO Should Be Able To Explain

For each case, CDO should be able to answer:

- Which service is affected?
- What is the incident timestamp and bounded query window?
- Is the case an outage, latency degradation, or noisy/ambiguous alert?
- Which metrics/logs/traces are hosted or exposed?
- Which records are primary RCAEval vs TF1 supplemental?
- Which Slack channel/Jira project/runbook would the report route to?
- Why the AI returns `DIAGNOSED` or `INVESTIGATE`.

## Minimal Demo Script For CDO

1. Open the evidence bundle for one case.
2. Show `time_window`, `metrics`, `logs`, `traces`, `ownership`, and `data_lineage`.
3. Show the corresponding adapted request in `datapack/external/adapted/`.
4. POST the adapted request to `/v1/triage`.
5. Open the report UI and show:
   - classification
   - confidence
   - suspected root cause
   - ranked RCA candidates
   - Slack-renderable raw response fields
   - Jira payload
   - audit id
