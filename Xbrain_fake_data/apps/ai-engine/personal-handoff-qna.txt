3. Architecture explanation

Short version:

Incident seed/event
-> CDO-hosted evidence bundle or read-only evidence proxy
-> TF1 AIOps context layer
-> POST /v1/triage
-> compute-first RCA
-> optional AgentCore synthesis
-> report + Slack payload + Jira ticket_payload

Longer version:

TF1 does not let AI directly call customer applications for logs or metrics. CDO/platform owns observability collection, storage, retention, auth, and bounded evidence access. AI Ops owns normalization, detection/context packaging, RCA, confidence gating, report generation, and Slack/Jira payload output.

The triage engine is event-driven. It is not continuously running expensive AI over every metric/log event. A lightweight detector/context layer identifies an incident candidate, collects bounded context, then calls `/v1/triage`.

6. What to say if mentor asks "Why not just send all logs/metrics into AI?"

Answer:

Because it is unsafe, expensive, and hard to audit. We only send bounded incident context:

- tenant scoped
- service scoped
- environment scoped
- time-window scoped
- redacted/sampled logs
- limited trace summaries
- metadata with audit IDs

The LLM, if enabled, only sees grounded evidence/report context after compute-first RCA. It does not receive unbounded raw telemetry or backend credentials.

7. What to say if mentor asks "What exactly does the AI do?"

Answer:

Current skeleton does compute-first triage:

- validates tenant/correlation headers
- validates request schema
- extracts metric/log/deploy/ownership features
- runs deterministic RCA classification
- computes confidence/status
- selects catalog-gated recommended actions
- returns report data, Slack payload, Jira ticket_payload, and audit_id

Optional AgentCore/LLM is only for grounded synthesis/investigation wording after deterministic evidence exists. It is not the source of truth for remediation.

11. What to say if CDO asks "Incident event cần chứa gì?"

Answer:

The incident seed/event only needs enough data to identify and scope the incident:

- schema_version
- tenant_id
- correlation_id
- incident_id
- environment
- service
- severity
- title/description
- started_at/received_at
- labels
- optional evidence_uri

It does not need to contain full metrics/logs. Full evidence can be hosted separately and looked up by incident_id/evidence_uri.

12. What to say if mentor asks "How do you measure MTTA/MTTR?"

Answer:

For W11, we document the baseline plan. For W12 final eval:

- MTTA reduction target: >= 50%
- MTTR reduction target: >= 30%, or time-to-actionable-ticket proxy if real fix time cannot be measured

Method:

Compare manual investigation steps versus Triage Hub flow on the same three scenarios:

- time to identify owner
- time to collect evidence
- time to produce suspected RCA
- time to create Jira payload/report
- time to notify Slack owner
